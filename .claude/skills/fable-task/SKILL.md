# /fable-task — 単発タスクの遂行

設計思想: **ミッションの重装備(状態ファイル 4 点)は省く。品質規律(委譲・検証・
正直な報告)は省かない。** タスクが小さいことは、雑にやってよい理由にならない。

## Step 0: 重量判定(最初の 10 秒)

| 判定 | 目安 | 進め方 |
|---|---|---|
| **軽量** | 単一ファイルの小修正・質問への回答・1 コマンドで済む調査 | 委譲せず指揮者が即実行。ただし検証(Step 3)は省かない |
| **中量** | 複数ファイルに触る・並列化できる独立作業がある・調査が要る | 分解 → 委譲(Step 1 へ) |
| **ミッション級** | セッションをまたぐ・数時間超え・完了定義が多段 | このスキルで受けず `/fable-mission` へ |

迷ったら軽い方で始めて、崩れたら昇格する。**昇格はいつでもできる。降格はできない。**

## Step 1: 分解と完了条件

- タスクを**独立して並列実行可能な単位**に分解する
- 各単位に**観測可能な完了条件**を書く。書けなければ理解が足りていない — 先に scout を出す
- 計画の事前提示は次の場合のみ: 破壊的操作を含む / 外部公開を伴う / 依頼の解釈が割れる。
  それ以外は提示せず進む(単発タスクに常時承認ゲートの重さは合わない。
  可逆で依頼の自然な延長なら動く — それが Fable の流儀)

## Step 2: 実行ループ

- ルーティング表(CLAUDE.md)どおりに委譲する: 調査 → scout / 実装 → builder /
  原因不明 → debugger。ブリーフは 5 要素 + 検証ハーネス(/fable-brief の手筋)
- 依存のない単位は**同一メッセージで並列委譲**。同時にファイルを変更する場合のみ worktree 分離
- builder が 2 回失敗 → debugger で診断してから再委譲。
  修正サイクルが 2 回で収束しなければユーザーに判断を仰ぐ(ミッションと同じ規律)

## Step 3: 検証ゲート(タスクの大小に関係なく省略禁止)

- コード変更は verifier の**動作検証**を通してから完了扱いにする(テスト green ≠ 動く)
- reviewer(Opus)は全タスクには課さない。変更が大きい・重要箇所(認証・決済・データ削除・
  公開 API)に触れた場合のみ追加する

## Step 4: 報告と後始末

- **結論から報告**: やったこと(✅ 検証済みかを明記)/ 判断したこと / 未解決・懸念
- 成長シグナル(修正・手戻り・驚き・摩擦・成功パターン)があれば
  `.fable-team/growth/inbox.md` に 1 行 — 単発タスクもチームの成長の材料である
- git リポジトリなら、区切りとしてコミットを提案する

## 昇格の判断(/fable-mission への切り替え)

次のサインが出たら、粘らずに昇格する:

- 見積もりが 2 倍を超えて崩れた(作業量ではなく前提の問題 — judgment playbook)
- このセッション内に終わらないことが確定した
- 作業中に完了条件が増殖し始めた(「ついでに」が 3 つを超えた)

昇格時は、ここまでの発見・判断・部分成果を `/fable-mission` の intake に持ち込む(捨てない)。
