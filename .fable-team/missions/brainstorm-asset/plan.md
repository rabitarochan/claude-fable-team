# 実行計画: ブレインストーミング支援資産の新設

> 作成: architect(2026-07-04)/ 承認: 未(ミッション承認ゲート待ち)
> 状態記号: ⬜ 未着手 / 🔄 進行中 / ✅ 完了(検証済み) / ⏸️ ブロック
> ✅ は verifier の挙動検証を通過したタスクのみ。

## 前提(この計画が成り立つ条件)

- **ミッション起票の承認ゲート(mission スキル Step 5)が、本計画の設計判断そのものへの発注者承認を兼ねる。** Phase 1 はその承認を前提に着手する(フレームワーク変更の事前承認要件を満たす)。
- **ブレストは単一セッションのプレ・ミッション活動**とする。セッションを跨いで練り続けたい場合、それ自体を mission に昇格させる(= 新しい永続的ブレスト状態ディレクトリは作らない)。
- 検証ハーネスは `claude plugin validate .` のみ(~2s)。スキルの**挙動**を測る自動テストは存在せず、挙動検証はドッグフーディング1回(Conductor-in-the-loop)で担保する。
- 新エージェント追加・マニフェスト変更・rules.md 変更は不要(scout 確認済み)。したがって consuming プロジェクトへの再 init 伝播も不要。
- ロールはプラグイン導入済みなら本来のエージェント、未導入なら general-purpose + ロールブリーフで代行する。

## 主要な設計判断

| 判断 | 採用案 | 理由 | 採らなかった案 |
|---|---|---|---|
| 資産の形 | **コマンドスキル(単一の手続き型 SKILL.md、`task` の兄弟)** | ブレストは「発散→収束→goal/DoD 草案化→intake へ受け渡し」という**手続きと下流ハンドオフ**が本質。task/mission と同じライフサイクル活動として起動される | playbook スキル(本質は途中適用のヒューリスティックでなく手続き。レンズカタログは Conductor がブリーフを組む道具で、サブエージェントが読む共有資料ではない → debug/verify を playbook 化する根拠が効かない)/ 新エージェント(着想は Conductor・architect の領分。固定モデル/ツールの反復ロールでなく agent のバーを満たさない)/ 2ファイルの hybrid(Playbooks 表の行 + 2つ目のファイルを増やすだけ = ブロート) |
| 発散エンジン | **Judge Panel / scout fan-out パターンを既存エージェント+レンズ別ブリーフで再利用。** 既定は Conductor 主導の対話、空間が広いときのみ並列レンズ fan-out | HANDOFF §5 の「異なる優先度で独立提案 → Opus が採点・統合」を着想生成に転用。既定を対話にして安く保つ(task↔mission のスケール分けと同型) | 専用 ideator エージェント(ブロート、固定モデル/ツール profile を持てない、着想は「ロール」でなく生成行為) |
| ライフサイクルフック | **`/fable-team:task` と `/fable-team:mission` の intake の上流。** 既存の task↔mission スケール分岐を延長 | 「ゴール自体があいまい」なときに発散を挟む一段を追加。intake の goal/DoD 確定を重複させない | ブレストで mission 4状態ファイルを生成(slug も mission もまだ無く時期尚早、mission Step 4 と重複)/ `.fable-team/brainstorms/` 新設(新しい列挙状態 = ブロート) |
| 出力アーティファクト | **可搬な「ブレスト・サマリ」ブロック**(問題枠づけ / 探索した選択肢 / 収束した方向 / **goal 草案 + 観測可能な DoD 草案** / 推奨 intake ルート) | intake にそのまま食わせられる最小形。mission 化時は mission.md/journal 起点エントリの素材になる。既定は ephemeral(永続ファイルを強制しない) | 必須の永続ファイル(ブロート) |
| 列挙点の差分 | **README 構造ツリー + CHANGELOG のみ** | 「足すとき、足さないものを探す」規律の実装 | Playbooks 表(playbook.md を持たない)/ Team Composition 表(agent 無し)/ plugin.json・marketplace.json(列挙なし)は**触らない** |

**再考の条件(reconsider if):**
- 進行手順の勘所+レンズカタログが単一ファイルに収まらなくなった、またはレンズ用サブエージェントが**パス渡しの共有参照**を必要とし始めたら → `playbook.md` を切り出し Playbooks 表に1行追加する(それまでは YAGNI)。
- ブレストが**固定モデル/ツール profile を持つ反復ロール**として頻出するようになったら → 専用エージェントに昇格。
- ユーザーがブレストをセッション跨ぎで保存・再訪したがるようになったら → 軽量な永続レコードを追加(ただしそれは大抵「それは mission だ」を意味する)。

## リスク

| リスク | 検知方法 | 顕在化したら |
|---|---|---|
| フル・デザイン思考フレームワークへのスコープ膨張(SCAMPER / Six Hats / Crazy 8s …) | SKILL.md が task 相当の単一ファイルを超える / レンズカタログの肥大 / reviewer が「kitchen sink」を指摘 | レンズは小さな核に限定。核以外の手法は grow バックログへ回し、ローンチには含めない |
| ブロート(冗長な列挙点) | reviewer が列挙差分を点検。README ツリー + CHANGELOG 以外に手が入っていないか | 新エージェント無し・新状態ディレクトリ無し・Playbooks 表無し・マニフェスト無し・単一ファイルを堅持 |
| 言語規則の逸脱(内部は英語だが、スキルが回す対話はユーザー言語) | reviewer が本文の非英語を grep / 「対ユーザーはユーザー言語」規則の明記を確認。2.1 で実対話が日本語だったか | スキル本文に規則を明記(rules.md 言語規則に倣う)し、ドッグフーディングで実地確認 |
| mission intake との役割重複・境界不明 | reviewer + 2.1/2.2 ドッグフーディング。「いつブレストし、いつ直接 intake するか」が判別できるか | 明快な境界を SKILL.md に記す。ブレスト = ゴール自体があいまいで発散が要るとき / intake = ゴール既知。intake の DoD 確定を「草案」以上に重複させない |
| ドッグフーディングが弱く、スキルの効果が判定不能 | 2.2 の受理検証(手戻りなく intake に採用できたか) | 本当にあいまいな実アイデアを選ぶ。薄ければそれ自体を指摘とし、Phase 2 ループでスキルを改訂 |
| 将来「専用 ideator エージェント」新設への圧力 | ビルド時は無し(設計判断に記録済み) | 上記「再考の条件」に照らして判断。反復ロール化して初めて昇格 |

## 補足(architect からの申し送り)

- **最小ファイルセット**: 追加は `skills/brainstorm/SKILL.md` の**1ファイルのみ**。`templates/` も `example/` も持たせない(出力サマリブロックは SKILL.md 内に fenced で埋め込む。mission の承認ブロックが SKILL.md 内に埋め込まれているのと同型)。これが「Bloat is death」に対する本ミッションの主張。
- **ドッグフーディングの題材**は本リポジトリ内の実在の改善候補(例: growth inbox の未処理シグナル、あるいは HANDOFF の未カバー領域)を推奨。自己適用が最も誠実な挙動検証になる。
